BUG-B-1Force-spawn heuristic disabled for Grok — every action verb misroutesCRITICAL
jarvis/brain/manager.py:966-971 · _should_force_openclaw
Symptom. When brain.primary is grok, openai, openrouter, or any non-Anthropic / non-Gemini provider, the deterministic spawn heuristic returns False unconditionally. Every utterance with an action verb (schreib, baue, öffne) bypasses the OpenClaw spawn path and falls through to LLM tool-choice — exactly the silent failure surface the heuristic was designed to prevent.

Root cause. A guard if primary not in ("claude-api", "gemini"): return False was added as a Gemini-403 workaround but never widened. The user's actual primary today is grok (per MEMORY.md BUG-017), so the guard kills the heuristic for the most common provider.

Fix direction. Gate on whether the configured OpenClaw worker is viable, not on a hardcoded provider whitelist. Allow any provider in _SECRET_KEY_TO_BRAIN whose worker class is callable.
If you say "build me a Python script" today, Jarvis is supposed to hand the job to its delegation worker. Instead it tries to answer with the chat model alone — because a half-finished safety check refuses to forward jobs for the provider you are actually using.
BUG-Z-1Whitelist matcher bypassable via LLM-controlled argument-key orderingCRITICAL
jarvis/safety/risk_tier.py:48-61 · _serialize_args
Symptom. The whitelist match string is f"{tool.name} {value1} {value2} …" — keys dropped, values joined in dict insertion order. A tool call like run-shell(prefix="ls", command="rm -rf /") serializes to "run-shell ls rm -rf /", matches "run-shell ls *" → tier downgraded to safe with approved_by="whitelist" → executes without confirmation and skips plausibility.

Root cause. Two-fold: keys are stripped from the matcher, and ordering follows attacker-controllable dict insertion. The whitelist also unconditionally downgrades to safe regardless of the tool's declared tier.

Fix direction. Match against canonical-form f"{tool.name} {k1}={v1} {k2}={v2}" with sorted keys, or pin the match to a "primary" argument per tool (e.g. args["command"] for shell).
The list of "you may run these without asking" patterns can be tricked. A model can hide a dangerous command behind a harmless-looking prefix and it slips past every safety prompt as if you had personally approved it.
BUG-S-2Voice death when _fanout task dies — heartbeat reports it, nothing actsBLOCKER
jarvis/speech/pipeline.py:1437-1448, 1517-1519 · _run_parallel_wake
Symptom. "Hey Jarvis" silently stops working after some uptime. The 10-second heartbeat correctly logs fanout=DEAD oww=alive whisper=alive — but asyncio.wait(detector_tasks) deliberately excludes the fanout task, both detector queues stop receiving chunks, and the function never returns. Same silent-failure class as BUG-014 / BUG-016.

Root cause. No try/except around mic.stream() iteration in _fanout, and the surrounding wait set does not include the fanout task. Heartbeat detects but cannot self-heal.

Fix direction. Include fanout_task in the asyncio.wait set, or have the heartbeat task call fanout_task.cancel() and raise from _run_parallel_wake when it sees fanout_alive=False.
The wake-word listener can quietly die mid-session. The logs see it die. No part of the code does anything about it. The user just notices Jarvis stops reacting.
BUG-P-10Gemini context-cache TTL-expiry wedge — root-cause comment exists, fix never codedBLOCKER
jarvis/plugins/brain/gemini.py:292-384
Symptom. One hour after Jarvis boot, the first voice turn hangs for forty seconds and ends with no audible reply. The Gemini server-side cache TTL expired; the client still sends the dead cache name; Gemini returns 403 CachedContent not found; BrainManager logs the provider as failed without invalidating self._cached_content_name; the next turn re-uses the dead name and re-fails.

Root cause. An in-file ninety-line comment block from 2026-05-11 documents the exact failure and prescribes the retry pattern — try / except PermissionDenied with "CachedContent not found" / set self._cached_content_name = None / re-add system_instruction + tools / single retry. The fix was never implemented.

Fix direction. Implement the in-file recipe. Keep recovery local to the Gemini provider; do not leak Gemini-specific semantics into BrainManager.
Once Jarvis has been on for an hour, the link to Gemini goes bad and stays bad until restart. The fix is already written down — in the source code as a comment — but nobody applied it.
BUG-X-2Cancelled missions keep spawning workers and burning budgetBLOCKER
jarvis/missions/kontrollierer/orchestrator.py:251-465, 859-875
Symptom. A POST /api/missions/{id}/cancel transitions the mission header to CANCELLED but the iteration loop has no state == CANCELLED early-exit. _safe_transition swallows IllegalStateTransition at debug level. The worker is spawned, the critic runs, budget.assert_under_limit still increments. There is no CancelToken threaded into iterations.

Root cause. Cancel is a header mutation only; no propagation into the running iteration body.

Fix direction. Add a self._cancel_event per mission; check it at the top of each iteration and before each _spawn_worker_collect. Wire POST /cancel to set() it.
Pressing "cancel" on a running task is cosmetic. The dashboard says it's cancelled, but the model in the background keeps running and keeps costing money until it finishes on its own.
BUG-W-2SessionRollupWorker never auto-fires — ADR-0014 "loud regression" silently brokenBLOCKER
jarvis/memory/wiki/session_rollup.py:329-348
Symptom. IdleDetector emits IdleEntered exactly once at the active→idle transition with idle_since_ns ≈ now − 5min. The worker compares (now − idle_since_ns) ≥ 120min — always false. The 120-minute auto-rollup never triggers; the session_rollups_succeeded / _failed counters stay at zero forever; the long-term memory tier silently does not exist.

Root cause. IdleEntered fires once at the boundary, not repeatedly during idle. Threshold semantics assume re-fire that never comes.

Fix direction. Either subscribe to a periodic "still-idle" tick, or compute the deadline as "fire-time + 120min" on the boundary event and arm a single asyncio timer.
The system was supposed to summarize a session into long-term memory after two hours of quiet. The trigger that should fire that summary never does — so nothing ever lands in long-term memory. The user has been running without that tier the entire time.
BUG-A-1AudioPlayer.stop() is a no-op — barge-in completely brokenBLOCKER
jarvis/audio/player.py:491-493
Symptom. Every barge-in / interrupt / announcement-priority path calls self._player.stop() expecting playback to abort. The implementation is sd.stop(), which by PortAudio contract only stops sd.play/sd.rec/sd.playrec — never persistent sd.OutputStreams, which is what the player uses. Jarvis keeps talking over the user; the code thinks it stopped.

Root cause. The stream is a closure-local in play_chunks and unreachable from stop(). The wrong API was wired.

Fix direction. Track the active stream as an instance attribute; on stop call stream.abort() (Pa_AbortStream, discards buffered samples).
When the user starts speaking during a Jarvis answer, Jarvis is supposed to stop. It doesn't. The "stop button" inside the code is wired to a switch that doesn't apply to how Jarvis actually plays audio.

BUG-K-10D9 recursion protection bypassed — run-skill can fork-bomb itselfCRITICAL
jarvis/ui/desktop_app.py:1078-1093 · jarvis/plugins/tool/run_skill.py:23-28
Symptom. The RunSkillTool docstring promises "D9 Recursion-Protection: SkillRunner is constructed without a tool_registry reference everywhere". Production code does the opposite — discovers every no-arg-constructible jarvis.tool entry-point and stuffs all into skill_tool_registry, including RunSkillTool itself. A SKILL body containing TOOL: run-skill {"skill_name": "self"} resolves and re-enters; no recursion-depth guard exists.

Fix direction. Explicitly remove run-skill from the skill_tool_registry dict at build time. Add a _depth counter to SkillRunner and refuse to nest deeper than two.
A skill can call itself in a loop with no limit. If anyone authors that — by accident or on purpose — the process crashes from stack overflow. The protection note in the source code claims this can't happen, but the code doesn't actually enforce it.

BUG-Y-2Drift-Guard auto-reverts every self-mod mutation within 5 minutesBLOCKER
scripts/drift-guard-daemon.ps1 · scripts/config-soll.json
Symptom. A successful voice-mutation writes tts.provider, audits OK, says "set" — then within 300 s the drift-guard daemon overwrites jarvis.toml from config-soll.json. Six of the eight ALLOWED self-mod paths (tts.provider, tts.voice_*, brain.primary, stt.provider) are in the daemon's enforcement set. Fix direction. Teach the drift-guard to update config-soll.json after acknowledged self-mod writes (single source of truth), or pause it during self-mod windows.

BUG-X-3Job-Object assign race — worker grandchildren escape kill-on-closeBLOCKER
jarvis/missions/workers/subjarvis_worker.py:529-542 · gemini_worker.py:314-327 · jarvis/missions/isolation/job_object.py:118-136
Symptom. The pattern create_subprocess_exec → set last_pid → job.assign(pid) leaves a window between process creation and Job Object assignment where the worker can already fork. Job Object membership is inherited at creation time, not retroactive — grandchildren escape. ADR-0009 §5 zombie-risk invariant violated.

Fix direction. Start the subprocess CREATE_SUSPENDED, assign to the Job, then ResumeThread. Standard Windows pattern.
When a heavy worker is killed, only the main process actually dies. Any helper processes the worker started keep running. After enough failures, the machine has a colony of orphans the orchestrator can no longer reach.

Cross-cutting patterns

§ themes
PATTERN 01

Silent-fallback anti-pattern

except Exception: pass followed by "pipeline keeps running in an inconsistent state". Ancestor of BUG-002 / 003 / 007 / 010 / 014. The audit found at least eight fresh occurrences across Brain, TTS, Safety, Config, and Audio.

PATTERN 02

Multi-layer drift

The five-layer parity test works for HANGUP_REASONS but the same discipline is absent for VoiceTier, mission statuses, skill lifecycle states, and the phase6.* Pydantic gap. Anti-drift scaffolding is local, not project-wide.

PATTERN 03

70%-built / 5%-wired

Phase-7 self-mod, Google STT, the skill system, and the wiki external-sources package all share the shape: code exists, tests exist, docs exist, nothing is reachable from production entry-points. The cost of a partial commit is invisible until production reads the dead reference.

PATTERN 04

Trust by naming

fnmatch substring matchers in safety, wake-priority, and device-blocklist all fail on unanchored or case-mismatched inputs. Each is individually recoverable; the class deserves a shared regex-anchor helper rather than per-call defensive code.
