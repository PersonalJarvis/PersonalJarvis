TTS phoneme-stretch localised & fixed via 5-agent deep-dive.

The user reported the TTS voice was elongating phonemes — "Hallo" coming out as "haaaaaaa lalala oooo". Five parallel Opus investigator agents converged on the same root cause: the AudioPlayer's "persistent" OutputStream was being closed and reopened on every sentence, causing WASAPI prebuffer + scipy.signal.resample_poly FIR ringing at every sentence boundary. The fix hoists the stream lifecycle to instance fields, makes stop() actually abort the active stream, and adds a success-log for future diagnosability. 8 tests green, 3-sentence live smoke-test passed end-to-end.

Symptom

§ what the user heard

The synthesized voice elongated individual phonemes. Instead of "Hallo" the user heard something closer to "haaaaaaaa lalala oooo" — extra L sounds, stretched vowels, the rhythm of speech smeared across sentence boundaries. The bug appeared at unpredictable cadence: sometimes one sentence sounded fine, the next one didn't. The model itself was not at fault — the audio bytes coming out of Gemini Flash TTS are 24 kHz mono PCM and are structurally correct. The corruption happened entirely on the playback side.

5-agent deep-dive · convergent findings

§ parallel opus investigators

Each agent received a different focused scope and ranked hypotheses with file:line citations. Four of five independently identified the per-sentence stream tear-down as the primary cause; the fifth provided runtime log evidence that turned the static analysis into a confirmed diagnosis.

A-01 · scope: jarvis/audio/player.py

Primary: asyncio.Lock from commit 33d51c5f serialises calls, but each play_chunks still tears down its OutputStream in finally. With streaming-TTS calling once per sentence, every sentence triggers scipy.signal.resample_poly polyphase FIR edge ringing.

Three stacked root causes

§ all real, all measured
LayerWhat was brokenHow it stretched audio
1 · Stream lifetime The finally block at player.py:487-489 closed sd.OutputStream at the end of every play_chunks call. The streaming-TTS pipeline calls play_chunks once per sentence, so every sentence opened a fresh stream — despite the docstring at player.py:419-424 explicitly designing for "one persistent stream per response". Every sentence boundary triggered WASAPI's latency='high' prebuffer (~200 ms) plus scipy.signal.resample_poly polyphase-FIR edge ringing at the new stream's start. Perceived as elongated, "lalala"-style phoneme duplication.
2 · Barge-in stop() player.py:491-493 implemented stop() as just sd.stop(). sd.stop() only affects streams opened via sd.play(); the persistent OutputStream instance was invisible to it. When an interrupt-priority announcement or background-completed event fired player.stop(), the old stream kept draining 200 ms of buffered audio while a new stream was already opened by the next producer. Two concurrent streams on the same WASAPI device produced phase-overlap artifacts.
3 · Diagnose-blindness The fallback cascade in _open_output_stream emitted a WARN on every failed rate but no success log on the chosen rate. Production log showed 193 failures, zero successes — invisible to diagnosis. Indirect: every previous attempt to debug this bug ran into "I see the cascade running, but which rate finally worked?" — no answer. Added a single log.info line so the next bug is observable.

The fix · three surgical edits

§ jarvis/audio/player.py · +67 / -27

The fix is surprisingly small — no pipeline changes, no API breakage, no model swap. The original architecture ("one persistent stream per response") was correct; the implementation had drifted into "one per sentence". Three edits restore the design intent. 

# 1 · __init__ — instance-scope stream lifecycle
+ self._active_stream: sd.OutputStream | None = None
+ self._active_source_rate: int | None = None
+ self._active_device_rate: int | None = None

# 2 · _open_output_stream — success log on the chosen rate
  stream.start()
+ log.info("OutputStream opened @ %d Hz (source=%d Hz, device=%s)",
+          target_rate, source_rate, self._device)
  return stream, target_rate

# 3 · play_chunks — _ensure_stream uses instance fields, NO finally close
  async with self._get_play_lock():
      def _ensure_stream(needed_rate):
+         if self._active_stream is not None and self._active_source_rate == needed_rate:
+             return self._active_stream, self._active_device_rate
+         if self._active_stream is not None:
+             self._close_output_stream(self._active_stream)
+             self._active_stream = None
          new_stream, device_rate = self._open_output_stream(needed_rate)
+         self._active_stream = new_stream
+         self._active_source_rate = needed_rate
+         self._active_device_rate = device_rate
          return new_stream, device_rate
      ...
      - finally:
      -     if stream is not None:
      -         await asyncio.to_thread(self._close_output_stream, stream)

# 4 · stop() — actually abort the persistent stream
  def stop(self) -> None:
+     stream = self._active_stream
+     self._active_stream = None
+     self._active_source_rate = None
+     self._active_device_rate = None
+     if stream is not None:
+         try: stream.abort()  # Pa_AbortStream — discards buffered samples
+         except Exception as exc: log.debug("OutputStream.abort() failed: %s", exc)
+         try: stream.close()
+         except Exception: pass
      sd.stop()

Documented follow-up

§ out of scope for this fix
FU-1 · WDM-KS error -9999 fallback During the live smoke test the auto-headset resolver picked a Windows WDM-KS device whose blocking-API is unimplemented (PaErrorCode -9999 'Blocking API not supported yet'). The existing fallback in _open_output_stream only catches -9997 (Invalid sample rate); -9999 is re-raised. This explains why Agent 5 saw 193 warnings in production but no success lines — the cascade never got past -9999 on certain devices. Suggested follow-up: broaden the except to also retry on -9999 with a different device index, or strengthen the WDM-KS filter in _resolve_output_device to also exclude WDM-KS devices that have no safer sibling (currently only excluded if a sibling exists).
FU-2 · Gemini 3.1 vs 2.5 rate handling Agent 02 flagged that gemini_flash_tts.py hardcodes GEMINI_TTS_SAMPLE_RATE = 24_000 for both the 3.1 primary and the 2.5 sibling-bridge fallback. Both currently emit 24 kHz, so it works — but if either model's output format changes, the chunk metadata would be wrong. Robust fix: parse part.inline_data.mime_type (format "audio/L16;codec=pcm;rate=NNNN"), use that rate on the AudioChunk. Cheap, future-proofing.
FU-3 · Pipeline-side: one play_chunks per turn Agent 04 noted that _brain_streaming still spawns one play_chunks per sentence. With the persistent-stream fix this no longer causes artifacts, but it does churn 5× through _ensure_stream's fast path and 5× through the asyncio-lock acquire/release per turn. A pipeline-side refactor that chains all sentence chunk iterators into a single play_chunks call would be tidier and shave a few hundred microseconds per turn. Not user-visible.

Confidence · High

WASAPI buffer growth / drain

B-02 · scope: stream lifecycle

Persistent stream + parallel-synthesis producer bursts caused buffer to grow. Inter-sentence wall-clock gaps then drained it to underrun.

Confidence · High

Pipeline = one play_chunks per sentence

B-03 · scope: pipeline._brain_streaming

Pipeline still spawns N play_chunks calls per turn. Persistence helps within a call but not across; the gap between calls drains the WASAPI buffer.

Confidence · High

Runtime log smoking gun

B-04 · scope: data/jarvis_desktop.log

"OutputStream opened" NEVER appears in 4.96 MB of log. Either the running process loaded a stale install (BUG-006 four-layer trap), or persistence isn't actually active. 17 fresh cascade-failures in the post-fix hour.

Confidence · High

stream.write() underflow flag discarded

B-05 · scope: _write_samples

stream.write() returns a bool reporting mid-write underrun. The bool was never captured → every underrun was invisible at log level. latency="high" reserves only ~10 ms on USB headsets.

Confidence · High

Three latent causes the Welle-1 fix had exposed

CauseMechanismFix in Welle 2
1 · latency="high" is a HINT On modern USB headsets like the AB13X used here, "high" resolves via default_high_output_latency = 0.010 s — only 10 ms of buffer. Any inter-sentence pipeline gap > 10 ms drains the buffer and PortAudio inserts a click. Replace with explicit latency=0.2 (200 ms). Matches LiveKit/Pipecat/RealtimeTTS canonical config. actual_latency=0.201s verified live.
2 · stream.write() underflow discarded PortAudio returns True when the write straddled an empty-buffer moment. The bool was thrown away — every underrun was silent at log level. Same silent-failure class as BUG-003 SAPI5 fallback. Capture the bool; emit WARNING PortAudio underflow during write (frames=…, source=…Hz, device=…Hz). Now visible.
3 · Per-turn samplerate cascade spam Pipeline calls player.stop() at end-of-turn (for barge-in). Stream torn down → next turn restarts the rate-cascade. AB13X rejects 24 kHz → 1× "naechste Rate" warning per turn. 17 warnings/hour. New _device_rate_cache: dict[source_rate → device_rate]. Once we learned 24 kHz fails on this device, subsequent opens skip straight to the cached working rate.
FU · install-freshness check Agent 4's evidence (zero "OutputStream opened" lines despite the fix being on disk since 23:09:14) suggests the running Jarvis loaded the OLD bytecode. The classic BUG-006 four-layer trap. Confirmed disk-side: jarvis.audio.player.__file__ resolves to the active tree, mtime is current, hasattr(AudioPlayer(), '_active_stream') is True. The user simply needs to restart from this tree to pick up Welle-1+2 fixes.
