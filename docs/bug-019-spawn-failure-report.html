BUG-019 — OpenClaw-Spawn schlägt immer fehl

TL;DR — zwei Schichten. Schicht 1 (Code, FIXED): Der Critic-Subprocess startete gar nicht — WinError 2, weil argv[0] als nackter String "gemini" übergeben wurde und Windows CreateProcess kein PATHEXT auswertet. Schicht 2 (Konto, USER-AKTION): Auch mit funktionierendem Subprocess gibt Gemini HTTP 429 „capacity exhausted" zurück — die Google-AI-Studio-Quota auf deinem Konto reicht nicht für den 3-Iteration-Critic-Loop (≈ 3–6 Calls pro Mission innerhalb 30 s).

Schicht 1: Code-Bug (gefixt)

Symptom

  • CriticRunner: subprocess returncode=1, stderr=''
  • critic failed: Critic lieferte zweimal keinen schema-validen JSON-Output
  • Nach 3 Iterationen MissionFailed reason='critic_loop_exhausted' → Voice sagt „Die OpenClaw-Mission ist fehlgeschlagen."

Wurzelursache

build_critic_cmd() baute argv mit "gemini" als argv[0]:

cmd = [
    "gemini",                # <-- nur Name, keine Extension, kein Pfad
    "--prompt", ...,
    "--model",  "gemini-3-flash-preview",
    ...
]
proc = await asyncio.create_subprocess_exec(*cmd, ...)

Auf Windows liegt das Binary als %APPDATA%\Roaming\npm\gemini.cmd. PowerShell findet's via PATHEXT, CreateProcess (Python's create_subprocess_exec) macht das nicht und wirft:

FileNotFoundError: [WinError 2] Das System kann die
                   angegebene Datei nicht finden

Diese Exception ging durch _invoke_once() hoch und wurde als generisches Subprocess-Fehler-Pattern geloggt — die echte Ursache war unsichtbar. Der Worker-Side hatte den Bug schon erkannt (_resolve_gemini_binary() im GeminiWorker), das Pendant beim Critic-Switch in Commit 3f7985a2 war übersehen.

Fix

Neuer Helper _resolve_cli_binary(name, fallback_exts) in jarvis/missions/critic/runner.py:

def _resolve_cli_binary(name, fallback_exts=()):
    import shutil
    found = shutil.which(name)
    if found:
        return found
    for ext in fallback_exts:
        found = shutil.which(name + ext)
        if found:
            return found
    return name        # last resort: bare name; let CreateProcess raise loud

Beide Code-Pfade (Claude und Gemini) routen argv[0] durch den Helper. Zusätzlich DEFAULT_TIMEOUT_SECONDS von 120 s → 240 s hochgesetzt (Gemini-Cold-Start braucht Context-Cache-Create).

Verifikation Schicht 1.
Manueller Subprocess-Repro mit aufgelöstem Pfad: EXIT=0, STDOUT_LEN=1236, response="{\"v\":\"ok\"}".
Resolved argv[0]: C:\Users\Administrator\AppData\Roaming\npm\gemini.CMD.
Tests: 166 / 166 grün (Critic + Injection + Agents).

Schicht 2: Quota auf deinem Gemini-Konto

Was die Logs zeigen

Nach Schicht-1-Fix hat der Critic-Subprocess wirklich gestartet — und nach 281 s mit folgendem Stderr abgebrochen:

cause: {
  code: 429,
  message: 'You have exhausted your capacity on this model.',
  details: [[Object], [Object]]
},
retryDelayMs: 10000

Gemini-CLI macht intern Exponential-Backoff (10 s → 20 s → 35 s → 35 s), nach 4 Retries gibt sie auf. Wirft den Job zurück an unsere CriticRunner — die kommt erstmals in den Timeout-Fall, dann Iter-1 neu spawnen → noch eine Critic-Call → wieder 429.

Warum das passiert

Google-AI-Studio Free-Tier hat strikte Per-Minute-Limits auf gemini-3-flash-preview (üblich: ~5–10 req/min). Phase-6 Critic-Loop ist designed für High-Quota-Anbieter:

IterWorker-CallCritic-Call 1Critic-Call 2 (reframe)
0GeminiGemini(nur bei JSON-Error)
1GeminiGemini(nur bei JSON-Error)
2GeminiGemini(nur bei JSON-Error)

Worst-Case 9 Gemini-Calls in < 60 s aus einem einzigen Mission-Spawn. Free-Tier killt das nach Call 5–6.

Was du tun kannst (User-Aktion)

  • Empfehlung: Billing auf aistudio.google.com aktivieren. Pay-as-you-go Flash kostet ~$0.075 / 1M Input-Tokens — eine Hello-World-Mission liegt bei ~$0.001. Damit verschwindet die Per-Minute-Quota komplett.
  • Alternative ohne Geld: ich kann den Critic-Loop auf der Gemini-Side aggressiver beschränken (z. B. nur 1 Call pro Iteration, kein adversarial-reframe-Retry). Halbiert die Quota-Last, könnte mit Free-Tier knapp reichen — kostet aber etwas Anti-Sycophancy-Schutz.
