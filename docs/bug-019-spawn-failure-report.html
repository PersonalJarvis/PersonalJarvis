BUG-019 — OpenClaw-Spawn always fails

TL;DR — two layers. Layer 1 (Code, FIXED): The Critic subprocess did not start at all — WinError 2, because argv[0] was passed as a bare string "gemini" and Windows CreateProcess does not evaluate PATHEXT. Layer 2 (Account, USER-ACTION): Even with a working subprocess, Gemini returns HTTP 429 "capacity exhausted" — the Google-AI-Studio quota on your account is insufficient for the 3-iteration Critic loop (≈ 3–6 calls per mission within 30 s).

Layer 1: Code Bug (fixed)

Symptom

  • CriticRunner: subprocess returncode=1, stderr=''
  • critic failed: Critic did not return schema-valid JSON output twice
  • After 3 iterations MissionFailed reason='critic_loop_exhausted' → Voice says "The OpenClaw mission has failed"

Root Cause

build_critic_cmd() built argv with "gemini" as argv[0]:

cmd = [
    "gemini",                # <-- name only, no extension, no path
    "--prompt", ...,
    "--model",  "gemini-3-flash-preview",
    ...
]
proc = await asyncio.create_subprocess_exec(*cmd, ...)

On Windows, the binary lives as %APPDATA%\Roaming\npm\gemini.cmd. PowerShell finds it via PATHEXT, but CreateProcess (Python's create_subprocess_exec) does not and throws:

FileNotFoundError: [WinError 2] The system cannot find the specified file

This exception propagated through _invoke_once() and was logged as a generic subprocess-error pattern — the root cause was invisible. The Worker-side had already detected the bug (_resolve_gemini_binary() in GeminiWorker), but the counterpart for the Critic-switch in commit 3f7985a2 was overlooked.

Fix

Neuer Helper _resolve_cli_binary(name, fallback_exts) in jarvis/missions/critic/runner.py:

def _resolve_cli_binary(name, fallback_exts=()):
    import shutil
    found = shutil.which(name)
    if found:
        return found
    for ext in fallback_exts:
        found = shutil.which(name + ext)
        if found:
            return found
    return name        # last resort: bare name; let CreateProcess raise loud

Both code paths (Claude and Gemini) route argv[0] through the helper. Additionally, DEFAULT_TIMEOUT_SECONDS was raised from 120 s to 240 s (Gemini cold start needs Context-Cache-Create).

Layer 2: Quota on your Gemini account

What the logs show

After Layer-1 fix, the Critic subprocess actually started — and after 281 s it terminated with the following stderr:

cause: {
  code: 429,
  message: 'You have exhausted your capacity on this model.',
  details: [[Object], [Object]]
},
retryDelayMs: 10000

Gemini-CLI internally does exponential backoff (10 s → 20 s → 35 s → 35 s), and after 4 retries it gives up. It returns the job to our CriticRunner — which hits the timeout case for the first time, then spawns iteration 1 again → another Critic call → 429 again.

Why this happens

Google-AI-Studio Free-Tier has strict per-minute limits on gemini-3-flash-preview (typical: ~5–10 req/min). The Phase-6 Critic-Loop is designed for high-quota providers:

IterWorker-CallCritic-Call 1Critic-Call 2 (reframe)
0GeminiGemini(only on JSON-Error)
1GeminiGemini(only on JSON-Error)
2GeminiGemini(only on JSON-Error)

Worst-case: 9 Gemini-Calls in < 60 s from a single mission spawn. Free-Tier kills that after call 5–6.

What you can do (User action)

  • Recommendation: Enable billing on aistudio.google.com. Pay-as-you-go Flash costs ~$0.075 / 1M input tokens — a hello-world mission costs around ~$0.001. This eliminates the per-minute quota entirely.
  • Alternative without money: I can restrict the Critic-Loop more aggressively on the Gemini side (e.g., only 1 call per iteration, no adversarial-reframe-retry). This halves the quota load and might barely work with free-tier — but costs some anti-sycophancy protection.
