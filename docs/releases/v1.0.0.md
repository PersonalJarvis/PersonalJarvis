**Personal Jarvis v1.0.0 — First public release** · 2026-06-29

Talk to your computer — and watch it do the work. Personal Jarvis is an open-source,
privacy-first voice agent that turns one spoken request into a fleet of self-checking AI
agents.

## Highlights

- **Voice-first pipeline** — wake word → speech-to-text → multi-provider Brain →
  text-to-speech, fully streaming, with honest, language-aware readbacks (`de` / `en` / `es`).
- **Provider-agnostic by design** — every tier (router, ack, STT, TTS, worker, critic)
  degrades or crosses provider families on a missing or dead key. No single provider is
  load-bearing, and credentials are managed entirely in-app.
- **Cross-platform core** — the base install boots on a headless `python:3.11-slim` VPS;
  Windows-desktop and local-voice features live behind optional extras.
- **Jarvis-Agents mission system** — isolated `git worktree` workers with a self-healing
  critic loop; every deliverable lands in **Outputs** as a downloadable artifact.
- **Plugin marketplace** (OAuth + MCP) and a cross-platform control CLI (`jarvisctl`).

### Also in this release

- Declared `click` as an explicit dependency so the `jarvisctl` CLI imports on a clean
  install, restoring a green CI.
- Restored `TRADEMARK.md` to the published tree and removed a dead documentation link.
- Stopped defaulting the archival store to the removed `chroma` backend.
- `pyproject` metadata rewritten for the public release: English, cross-platform
  description and a `[project.urls]` block.

## Install

One command on Windows, macOS, or Linux — no Docker, no Python-version archaeology. Bring
your own keys; nothing is bundled.

**Windows** (PowerShell)

```powershell
irm https://raw.githubusercontent.com/PersonalJarvis/PersonalJarvis/main/install/install.ps1 | iex
```

**macOS · Linux**

```bash
curl -fsSL https://raw.githubusercontent.com/PersonalJarvis/PersonalJarvis/main/install/install.sh | bash
```

## Verify the release (recommended)

Releases are signed and can be verified before you trust them:

- **cosign / TUF** — the signed-release chain lives in [`install/`](../../install/): run
  [`install/install-verify.sh`](../../install/install-verify.sh) (macOS/Linux) or
  [`install/install-verify.ps1`](../../install/install-verify.ps1) (Windows) to check the
  signature and TUF metadata against the pinned trust root in
  [`install/TRUST_ROOT.md`](../../install/TRUST_ROOT.md) before running anything.
- Prefer to read first? The installers only create a venv, install dependencies, and run the
  first-run wizard — every line is open source.

## Maintainer checklist — how to publish this on GitHub

These are the exact clicks. Nothing here is part of the public release text above.

1. Open the repository on GitHub → in the right sidebar click **Releases** → **Draft a new
   release** (top-right button).
2. **Choose a tag** → type `v1.0.0` → click **Create new tag: v1.0.0 on publish**. Leave
   **Target** on `main`.
3. **Release title:** `v1.0.0 — First public release`.
4. **Describe this release:** copy everything from the top of this file down to the `---`
   line and paste it into the body box.
5. Leave **Set as a pre-release** unchecked. Tick **Set as the latest release**.
6. *(Optional)* If you produced signed build artifacts, drag them into the
   **Attach binaries** area so the cosign/TUF verification above has something to check.
7. Click **Publish release**.

> Note: the `v1.0.0` git tag does not exist yet — step 2 creates it on publish. The current
> newest tag in the repo is `v0.13.0`; `pyproject.toml` and `CHANGELOG.md` already declare
> `1.0.0`, so publishing this release is what makes the tag match the version.
