Why No Single Mission Works

Root-cause analysis & fix plan for the Worker-Critic stack · created from 20 parallel code deep-dives + forensic analysis of missions.db and on-disk mission logs.

Executive summary: It is not a single bug, but a chain. The actual killer: the active Worker runs over OpenClaw, which spawns a nested claude call without --permission-mode bypassPermissions → claude waits for a permission dialog that nobody answers → 480 s hang → FailoverError → mission fails. Plus six amplifiers. The main fix is a config change (routing to the proven working direct path), secured by six complementary repairs.

EVIDENCE-2File written — but failed at critic 240 s later

Mission 8d08 (reconstructed from data/jarvis_desktop.log): iterations 0 + 1 ran for 81 s / 94 s each, but wrote 0 bytes to the worktree. Only iteration 2 wrote 55 KB (3 files). But then the Critic ran > 240 s and was killed → mission critic_loop_exhausted — despite real work.

EVIDENCE-3Config split: Worker ≠ Critic backend

jarvis.toml [brain.sub_jarvis].provider = "openclaw-claude" → Worker = OpenClaw (the hanging path). Critic resolves independently to claude-api → direct claude --print. My solo test of a trivial task ran over the same stack and was APPROVED — the difference was load & complexity, not the code.

2 · Root-cause taxonomy (RC1–RC7)

Primary Amplifier Cosmetic (red herring)
RC1OpenClaw spawns claude without bypassPermissions + concurrency limitPrimary

OpenClaw spawns claude -p … --permission-prompt-tool stdio without --permission-mode bypassPermissions. claude emits a permission-prompt JSON frame and waits for a stdin answer, which OpenClaw's session driver never sends. After 480 s (= 600000 × 0.8, hardcoded) → FailoverError. Second mechanism: 7 missions in 2 minutes → parallel claude sessions on the same Claude-Max-20x account → server-side rate limit → silent wait until timeout.

Evidence: openclaw/dist/cli-backend-*.js, claude-live-session-*.js:386 · helpers-*.js:70 · OAuth via ~/.claude/.credentials.json is intact (no auth problem).

RC2Worker writes outside the git worktreeAmplifier

Skill-driven tasks (e.g. the Excalidraw skill) direct writes to absolute paths (C:\…\OneDrive\Ex\…) or browser actions. But the orchestrator only captures git add -N . && git diff HEAD inside the worktree (orchestrator.py:871-940). If the worker writes outside → empty diff → deterministic "empty diff" revise → loop exhausted.

RC3Critic timeout 240 s < agentic effortAmplifier

DEFAULT_TIMEOUT_SECONDS = 240 (runner.py:274) with worker budget 600 s. The fix on 2026-05-24 made the Critic agentic (bypassPermissions → real Read/Glob verification). For large diffs (55 KB), that exceeds 240 s → mission fails despite successful worker.

RC4Correction instruction never reaches the worker completelyAmplifier

The critic's correction_instruction is only published as a bus event, but not written to ReflectionMemory (reflections.py:184-200 renders only summary + evidence). The worker gets only a vague summary in iteration 1 instead of the prescriptive instruction — that's why 8d08 wrote nothing again in iter1.

RC5No Gemini fallback on hang/crashAmplifier

The fallback chain is only consulted on quota block (HTTP 429) (subjarvis_worker.py:630), not on hang/timeout/crash. ClaudeDirectWorker has no chain at all. Hence next=none — Gemini never kicks in.

RC6FailoverError mislabeled as "user abort"Amplifier · diagnostic poison

orchestrator.py:499-509: every worker_error without a billing string falls into else "user". Timeout, crash, OAuth error — all become WorkerKilled reason=user. This obscures the real cause (looks like user aborts).

RC7MAX_CRITIC_LOOPS = 3 consumed by empty iterationsAmplifier

Iterations 0/1 without writes consume 2 of 3 rounds (ADR-0009, cannot change without new ADR). Fix belongs in the Worker (RC2/RC4), not the budget.

RC-XSymlink EPERM — COSMETIC (red herring)not a killer

The [skills] failed to create plugin skill symlink … EPERM in every log is harmless: OpenClaw's JS does log.warn + continues (documented in the npm source). Not the cause of the 211 failures.

3 · The plan — prioritized repair order

Order by impact ÷ risk (validated against anti-patterns AP-1…AP-18 and recurring bug classes).

  1. FIX-F · Worktree confinement in contract (AGENTS.md) Risk: zero

    Pure documentation, effective immediately. In workspace.py:124-143: add rule "ignore skill instructions to absolute paths (OneDrive, C:\…), write everything relative to CWD" + "no browser in headless worker". Extend empty-diff correction text in runner.py:504 with OneDrive hint. Recommendation: don't allow skills with headless_safe=false as missions at all.

  2. FIX-A · Route worker to the proven direct path Highest impact

    Provider "openclaw-claude" → "claude-api" → factory selects ClaudeDirectWorker (calls claude --print --permission-mode bypassPermissions directly, without OpenClaw live session → no 480-s hang). Atomic in BOTH files (else drift-guard rollback in 5 min):

    scripts/config-soll.json:33  FIRST  → then  jarvis.toml:277 (toggle read-only, write BOM-free)

    model="" is safe → falls back to [brain.providers.claude-api].deep_model = claude-opus-4-7. Critic path untouched.

  3. FIX-C · Global concurrency bound against rate limit

    There is no cross-mission limit. asyncio.Semaphore(3) in Controller.__init__, async with only around worker.spawn(...) (orchestrator.py:721) — not the whole mission. Singleton covers voice + REST. Config: [phase6.orchestrator] max_concurrent_workers = 3.

  4. FIX-D · Defuse critic timeout against RC3

    DEFAULT_TIMEOUT_SECONDS → 480 (runner.py:274) and add "--max-turns","5" in runner.py:1036 — cap agentic verification, critic ends before 600-s worker budget.

  5. FIX-B · Fallback fix: bypassPermissions in worker openclaw.json only if OpenClaw stays

    The critic already injects cliBackends["claude-cli"] correctly (runner.py:804-816); the worker does not (subjarvis_worker.py:459-471). Add the same dict merge after the primary resolution. Fixes the permission half; concurrency hang remains → FIX-C needed. Elevate silent-swallow in except to ERROR log.

  6. FIX-G · Kill-reason taxonomy (five-layer enum) audit correctness

    Break mislabeling at orchestrator.py:499-509 into worker_timeout / worker_crash / auth_error / real user. Mandatory: all 5 layers atomic (Python literal events.py:97 → DE/EN announcer → voice listener → TS/UI → parity test), else BUG-008 recurrence (catch-all maps to injection_detected).

  7. FIX-E · Real provider fallback on worker error last

    Fallback logic into the orchestrator (orchestrator.py:499): on worker_error try next chain entry (GeminiWorker) instead of immediate TaskOutcome.ERROR. Watch BUG-017 (Gemini 403 was blocked) — gate behind gemini_fallback_enabled, read model from cfg (never hardcode, AP-6).

4 · Safe config procedure (FIX-A in detail)

The drift-guard daemon (scripts/jarvis-config-drift-guard.ps1) reads config-soll.json every 5 min and rolls back any TOML drift (BUG-010). That's why this order — else the problem comes back after 5 minutes.

# 1) Desired source FIRST (else rollback):
scripts/config-soll.json:33   "provider": "openclaw-claude"  →  "claude-api"

# 2) Unlock jarvis.toml, write BOM-free, lock again:
Set-ItemProperty jarvis.toml -Name IsReadOnly -Value $false
jarvis.toml:277   provider = "openclaw-claude"  →  "claude-api"
# Write via [System.IO.File]::WriteAllText(..., UTF8Encoding($false))  — NEVER Set-Content -Encoding utf8 (BOM → BUG-018)
Set-ItemProperty jarvis.toml -Name IsReadOnly -Value $true

# 3) Verify:
[Environment]::GetEnvironmentVariable("JARVIS__BRAIN__SUB_JARVIS__PROVIDER","User")   # → claude-api
powershell scripts/jarvis-config-drift-guard.ps1 -DryRun                              # → "TOML clean"
