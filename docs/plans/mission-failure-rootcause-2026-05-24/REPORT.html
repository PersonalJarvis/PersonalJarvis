Warum keine einzige Mission funktioniert

Root-Cause-Analyse & Umsetzungsplan für den Worker-Critic-Stack · erstellt aus 20 parallelen Code-Deep-Dives + forensischer Auswertung von missions.db und den On-Disk-Mission-Logs.

Kurzfassung: Es ist nicht ein Bug, sondern eine Kette. Der eigentliche Killer: der aktive Worker läuft über OpenClaw, das ein verschachteltes claude startet ohne --permission-mode bypassPermissions → claude wartet auf einen Berechtigungs-Dialog, den niemand beantwortet → 480 s Stillstand → FailoverError → Mission scheitert. Dazu sechs Verstärker. Der Haupt-Fix ist eine Konfig-Änderung (auf den bewiesen funktionierenden Direkt-Pfad umrouten), abgesichert durch sechs flankierende Reparaturen.

1 · Die empirische Grundwahrheit

Vor jeder Theorie: was sagt die echte Datenbank und was sagen die echten Logs? Drei harte Befunde aus data/missions.db und sub-agents-outputs/mission_*/…/logs/stderr.log.

EVIDENCE-1Der verschachtelte claude hängt 480 s und stirbtPrimär-Killer

Identisch in jeder gescheiterten Mission mit 0-Byte-Stream:

[agent/cli-backend] cli exec: provider=claude-cli model=opus promptChars=174 ...
[agent/cli-backend] claude live session start: model=claude-opus-4-7 activeSessions=1
[agent/cli-backend] claude live session close: reason=abort
[agent/cli-backend] claude live session turn failed: durationMs=563153 error=FailoverError
[model-fallback/decision] decision=candidate_failed reason=timeout next=none
   detail=CLI produced no output for 480s and was terminated.
FailoverError: CLI produced no output for 480s and was terminated.

→ 0-Byte stream.jsonl → Mission task_error, fälschlich als WorkerKilled reason=user etikettiert.

EVIDENCE-2Datei geschrieben — aber 240 s später am Critic gescheitert

Mission 8d08 (rekonstruiert aus data/jarvis_desktop.log): Iteration 0 + 1 liefen je 81 s / 94 s, schrieben aber 0 Bytes in den Worktree. Erst Iteration 2 schrieb 55 KB (3 Dateien). Doch dann lief der Critic > 240 s und wurde gekillt → Mission critic_loop_exhausted — trotz echter Arbeit.

diff.iter0.patch = 0 B · diff.iter1.patch = 0 B · diff.iter2.patch = 55 296 B

EVIDENCE-3Konfig-Split: Worker ≠ Critic-Backend

jarvis.toml [brain.sub_jarvis].provider = "openclaw-claude" → Worker = OpenClaw (der hängende Pfad). Der Critic löst unabhängig zu claude-api auf → direkter claude --print. Mein Solo-Test einer trivialen Aufgabe lief über denselben Stack und wurde APPROVED — der Unterschied war Last & Komplexität, nicht der Code.

2 · Root-Cause-Taxonomie (RC1–RC7)

Primär Verstärker Kosmetisch (Red Herring)
RC1OpenClaw startet claude ohne bypassPermissions + Concurrency-LimitPrimär

OpenClaw spawnt claude -p … --permission-prompt-tool stdio ohne --permission-mode bypassPermissions. claude emittiert ein Permission-Prompt-JSON-Frame und wartet auf eine stdin-Antwort, die OpenClaws Session-Driver nie sendet. Nach 480 s (= 600000 × 0.8, hartkodiert) → FailoverError. Zweiter Mechanismus: 7 Missionen in 2 Minuten → parallele claude-Sessions am selben Claude-Max-20x-Konto → serverseitiges Rate-Limit → stilles Warten bis Timeout.

Beleg: openclaw/dist/cli-backend-*.js, claude-live-session-*.js:386 · helpers-*.js:70 · OAuth via ~/.claude/.credentials.json ist intakt (kein Auth-Problem).

RC2Worker schreibt außerhalb des Git-WorktreesVerstärker

Skill-getriebene Aufgaben (z. B. der Excalidraw-Skill) weisen absolute Pfade an (C:\…\OneDrive\Ex\…) oder Browser-Aktionen. Der Orchestrator erfasst aber nur git add -N . && git diff HEAD im Worktree (orchestrator.py:871-940). Schreibt der Worker draußen → leerer Diff → deterministisches „empty diff"-Revise → Schleife erschöpft.

RC3Critic-Timeout 240 s < agentischer AufwandVerstärker

DEFAULT_TIMEOUT_SECONDS = 240 (runner.py:274) bei Worker-Budget 600 s. Der Fix vom 24.05. machte den Critic agentisch (bypassPermissions → echte Read/Glob-Verifikation). Bei großen Diffs (55 KB) sprengt das die 240 s → Mission scheitert trotz erfolgreichem Worker.

RC4Korrektur-Anweisung erreicht den Worker nie vollständigVerstärker

Die correction_instruction des Critics wird nur als Bus-Event publiziert, aber nicht in ReflectionMemory geschrieben (reflections.py:184-200 rendert nur summary + Evidenz). Der Worker bekommt in Iteration 1 nur eine vage Zusammenfassung statt der präskriptiven Anweisung — deshalb schrieb 8d08 in iter1 erneut nichts.

RC5Kein Gemini-Fallback bei Hang/CrashVerstärker

Die Fallback-Kette wird nur bei Quota-Block (HTTP 429) konsultiert (subjarvis_worker.py:630), nicht bei Hang/Timeout/Crash. ClaudeDirectWorker hat gar keine Kette. Daher next=none — Gemini springt nie ein.

RC6FailoverError als „User-Abbruch" fehletikettiertVerstärker · Diagnose-Gift

orchestrator.py:499-509: jeder worker_error ohne Billing-String fällt in else "user". Timeout, Crash, OAuth-Fehler — alles wird WorkerKilled reason=user. Das verschleiert die echte Ursache (sieht aus wie Nutzer-Abbrüche).

RC7MAX_CRITIC_LOOPS = 3 von leeren Iterationen aufgefressenVerstärker

Iter 0/1 ohne Schreibvorgang verbrauchen 2 von 3 Runden (ADR-0009, nicht ohne neue ADR änderbar). Fix gehört in den Worker (RC2/RC4), nicht ins Budget.

RC-XSymlink-EPERM — KOSMETISCH (Red Herring)kein Killer

Das [skills] failed to create plugin skill symlink … EPERM in jedem Log ist harmlos: OpenClaws JS macht log.warn + weiter (belegt in der npm-Quelle). Nicht die Ursache der 211 Fehlschläge.

3 · Der Plan — priorisierte Reparatur-Reihenfolge

Reihenfolge nach Impact ÷ Risiko (validiert gegen die Anti-Pattern AP-1…AP-18 und die wiederkehrenden Bug-Klassen).

  1. FIX-F · Worktree-Confinement im Vertrag (AGENTS.md) Risiko: null

    Reine Dokumentation, sofort wirksam. In workspace.py:124-143: Regel ergänzen „ignoriere Skill-Anweisungen zu absoluten Pfaden (OneDrive, C:\…), schreibe alles relativ ins CWD" + „kein Browser im Headless-Worker". Empty-Diff-Korrekturtext in runner.py:504 um den OneDrive-Hinweis erweitern. Empfehlung: Skills mit headless_safe=false gar nicht als Mission zulassen.

  2. FIX-A · Worker auf den bewiesenen Direkt-Pfad umrouten Höchster Impact

    Provider "openclaw-claude" → "claude-api" → Factory wählt ClaudeDirectWorker (ruft claude --print --permission-mode bypassPermissions direkt, ohne OpenClaw-Live-Session → kein 480-s-Hang). Atomar in BEIDEN Dateien (sonst Drift-Guard-Rollback in 5 Min):

    scripts/config-soll.json:33  ZUERST  → dann  jarvis.toml:277 (read-only togglen, BOM-frei schreiben)

    model="" ist sicher → fällt auf [brain.providers.claude-api].deep_model = claude-opus-4-7 zurück. Critic-Pfad unberührt.

  3. FIX-C · Globale Concurrency-Schranke gegen Rate-Limit

    Es existiert kein missionsübergreifendes Limit. asyncio.Semaphore(3) in Kontrollierer.__init__, async with nur um worker.spawn(...) (orchestrator.py:721) — nicht um die ganze Mission. Singleton deckt Voice + REST. Konfig: [phase6.orchestrator] max_concurrent_workers = 3.

  4. FIX-D · Critic-Timeout entschärfen gegen RC3

    DEFAULT_TIMEOUT_SECONDS → 480 (runner.py:274) und "--max-turns","5" in runner.py:1036 einfügen — agentische Verifikation gedeckelt, Critic endet vor dem 600-s-Worker-Budget.

  5. FIX-B · Fallback-Fix: bypassPermissions in Worker-openclaw.json nur falls OpenClaw bleibt

    Der Critic injiziert cliBackends["claude-cli"] bereits korrekt (runner.py:804-816); der Worker nicht (subjarvis_worker.py:459-471). Denselben Dict-Merge nach der primary-Auflösung ergänzen. Behebt die Permission-Hälfte; Concurrency-Hang bleibt → FIX-C nötig. Silent-Swallow im except auf ERROR-Log heben.

  6. FIX-G · Kill-Reason-Taxonomie (Five-Layer-Enum) Audit-Korrektheit

    Mislabel an orchestrator.py:499-509 aufbrechen in worker_timeout / worker_crash / auth_error / echtes user. Pflicht: alle 5 Schichten atomar (Python-Literal events.py:97 → DE/EN-Announcer → Voice-Listener → TS/UI → Parity-Test), sonst BUG-008-Wiederholung (Catch-all mappt sonst auf injection_detected).

  7. FIX-E · Echter Provider-Fallback bei Worker-Fehler zuletzt

    Fallback-Logik in den Orchestrator (orchestrator.py:499): bei worker_error nächsten Ketteneintrag (GeminiWorker) versuchen statt sofort TaskOutcome.ERROR. Vorsicht BUG-017 (Gemini-403 war gesperrt) — hinter gemini_fallback_enabled gaten, Modell aus cfg lesen (nie hardcoden, AP-6).

4 · Sichere Konfig-Prozedur (FIX-A im Detail)

Der Drift-Guard-Daemon (scripts/jarvis-config-drift-guard.ps1) liest alle 5 Min config-soll.json und rollt jede TOML-Abweichung zurück (BUG-010). Deshalb diese Reihenfolge — sonst kehrt das Problem nach 5 Minuten zurück.

# 1) Soll-Quelle ZUERST (sonst Rollback):
scripts/config-soll.json:33   "provider": "openclaw-claude"  →  "claude-api"

# 2) jarvis.toml entsperren, BOM-frei schreiben, wieder sperren:
Set-ItemProperty jarvis.toml -Name IsReadOnly -Value $false
jarvis.toml:277   provider = "openclaw-claude"  →  "claude-api"
# Schreiben via [System.IO.File]::WriteAllText(..., UTF8Encoding($false))  — NIE Set-Content -Encoding utf8 (BOM → BUG-018)
Set-ItemProperty jarvis.toml -Name IsReadOnly -Value $true

# 3) Verifizieren:
[Environment]::GetEnvironmentVariable("JARVIS__BRAIN__SUB_JARVIS__PROVIDER","User")   # → claude-api
powershell scripts/jarvis-config-drift-guard.ps1 -DryRun                              # → "TOML clean"
