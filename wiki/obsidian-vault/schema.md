# Wiki Schema ‚Äî Personal Jarvis Knowledge Vault

This file is the contract that every LLM editing this vault MUST follow.
It is read at the top of every wiki write operation. If a rule here
contradicts an in-page instruction, this file wins.

Inspired by Andrej Karpathy's *LLM Wiki* pattern (gist 442a6bf). The
core idea: knowledge is **compiled once and maintained continuously**,
not retrieved-and-rederived on every query. The LLM is the editor; the
markdown files are the codebase; Obsidian (or the Jarvis Desktop App
Wiki section) is the IDE.

## Three-Tier Memory

This vault is the long-term tier. It coexists with two faster layers
that live elsewhere ‚Äî do not duplicate their content.

| Tier | Lifespan | Lives in | Read by |
|---|---|---|---|
| Short-term | seconds to 30 min | `AwarenessManager.state` (RAM) | `awareness-snapshot` tool |
| Mid-term | last 3-5 sessions | `data/workspace/sessions/*.md` (this vault) | system-prompt injector |
| Long-term | permanent | `data/workspace/{entities,concepts,projects}/*.md` (this vault) | `wiki-recall` tool, system-prompt injector |

Short-term episodes feed mid-term session rollups. Mid-term session
rollups feed long-term entity/concept/project pages. The flow is one
way: information flows from fast/volatile to slow/permanent. Never the
other way around.

## Page Types

### Entity page (`entities/<slug>.md`)

Concrete singular things. One file per entity. Slug is kebab-case,
ASCII, derived from the entity's canonical name.

Required frontmatter:

```yaml
---
type: entity
entity_kind: person | tool | repository | service | device
slug: <slug>
aliases: [list, of, alt, names]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

Required body sections (in this order, do not rename):
1. `# <Canonical Name>` ‚Äî H1, never abbreviated
2. `## Summary` ‚Äî 2-4 sentences, the elevator pitch
3. `## Facts` ‚Äî bulleted facts with citations to `log.md` entries when possible
4. `## Relationships` ‚Äî `[[wikilinks]]` to related entities/concepts/projects with one-line explanation
5. `## Sources` ‚Äî backlinks to `log.md` entries and `attachments/` files

### Concept page (`concepts/<slug>.md`)

Abstract recurring ideas (architectures, patterns, methodologies).

Required frontmatter:

```yaml
---
type: concept
slug: <slug>
aliases: [list]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

Body sections: `Summary`, `Definition`, `Examples`, `Related`, `Sources`.

### Project page (`projects/<slug>.md`)

Active or recently-active workstreams. A project that has been dormant
for 90+ days should be moved to `_archive/projects/`.

Required frontmatter:

```yaml
---
type: project
slug: <slug>
status: active | paused | done | archived
started: YYYY-MM-DD
last_activity: YYYY-MM-DD
---
```

Body sections: `Goal`, `Current Status`, `Recent Activity`, `Open Threads`,
`Related`, `Sources`.

### Session rollup (`sessions/<YYYY-MM-DD>-<id>.md`)

Mid-term tier ‚Äî auto-generated by the `SessionRollupWorker` at session
end. Synthesises the L2 episodes that occurred during a single session
into one digest.

Required frontmatter:

```yaml
---
type: session
date: YYYY-MM-DD
started_at: HH:MM
ended_at: HH:MM
episode_ids: [42, 43, 44, ‚Ä¶]    # SQLite awareness_episodes rowids
session_id: <short-id>
---
```

Body: free-form prose, max 400 words, must reference relevant entities
and projects via `[[wikilinks]]`. Older than 7 days ‚Üí move to `_archive/sessions/`.

## Wikilinks

Use double-bracket syntax `[[entities/the maintainer]]` or short form `[[the maintainer]]`
when the slug is globally unique. Aliased links `[[entities/the maintainer|the user]]`
are allowed but discouraged ‚Äî prefer adding the alias to the entity's
frontmatter and using the canonical short form.

A broken wikilink is a bug. The wiki curator MUST either create the
missing page during the same ingest, or refuse the link and use plain
text. Never leave dangling `[[]]`.

## The `log.md` File

Append-only chronological record. Every ingest, every entity creation,
every concept extraction adds one entry. Format:

```
## [YYYY-MM-DD HH:MM] <verb> | <subject>
- pages touched: [[entities/x]], [[concepts/y]]
- source: <one-sentence description, link to attachments/ if applicable>
- summary: <2-3 sentences about what changed>
```

Verbs: `ingest`, `update`, `create`, `merge`, `rename`, `archive`, `delete`.

Never edit existing log entries. Never sort them ‚Äî chronology is part of
the data. `grep "^## \[" log.md | tail -5` must always return the five
most recent entries.

## Write Discipline (the hard part)

When ingesting a single source, the curator typically touches 10-15
pages. The discipline is:

1. **Read the source.** Form a mental list of entities, concepts, and
   projects it references or affects.
2. **Read the existing pages for each referenced item.** A new fact
   often updates several pages.
3. **Make the smallest correct edit per page.** Do not rewrite a page
   wholesale when adding one fact.
4. **Maintain cross-references.** If a fact connects entity A to
   entity B, both pages get an entry in their `Relationships` section.
5. **Add one log entry covering all the edits in this ingest.**
6. **Update `index.md` only when a new page is created**, not on every edit.
7. **Run a consistency pass.** Are all wikilinks valid? Are
   `updated:` dates current? Did any page exceed 500 lines (split it)?

## Hard Negatives

- Never duplicate content between pages. Use wikilinks.
- Never use vector embeddings as the primary index. FTS5 + structured
  wikilinks is the canonical retrieval path.
- Never mutate `log.md` past the most recent entry.
- Never store secrets, API keys, or credentials in any page.
- Never write to `_archive/` unless explicitly archiving an active page.
- Never let a page exceed 500 lines ‚Äî split into sub-pages with
  wikilinks.
- Never rename a slug without updating every backlink. The curator
  rename verb is a multi-step operation: rename file, update all
  backlinks, append log entry with the rename verb.

## Scale Targets

- Total vault: target 100-500 pages over the first year.
- Per-page: 50-300 lines is the sweet spot. 500 lines is the hard cap.
- Per-session-rollup: 200-400 words.
- `index.md`: kept under 200 lines via category headings.

## Versioning

The vault is a git repo. Every meaningful curator run produces one
commit with the message `wiki: <verb> <subject>` and the body referencing
the `log.md` entry. Manual edits by the human user are committed
separately with `wiki(manual): <description>`.
